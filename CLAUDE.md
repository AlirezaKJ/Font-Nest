# FontNest working agreement

FontNest is a Tauri 2 desktop font manager: Rust backend (`src-tauri/`), SvelteKit 5 + Tailwind 4
frontend (`src/`), typed IPC bindings generated by ts-rs (`src/lib/bindings/`).

## Always keep the changelog current

`CHANGELOG.md` (repo root) is more than docs. The app reads it at runtime to render the in-app
What's New view (`src/lib/components/PatchNotesView.svelte`, parser in
`src/lib/release-notes/changelog.ts`). Keeping it current is part of finishing any change, not a
separate chore.

Whenever you make a user-facing change, add or update an entry under `## [Unreleased]` in
`CHANGELOG.md` as part of the same change, before you consider the work done. Treat it like
updating tests: a user-facing diff without a changelog entry is incomplete.

- User-facing = anything a person using FontNest could notice: new features, changed behavior or
  UI, fixed bugs, removed functionality, security fixes.
- **Skip** purely internal work with no user-visible effect (refactors, test-only changes,
  formatting, dependency bumps, CI). When in doubt, ask whether a user would notice; if yes,
  write an entry.
- Write for the user, in plain language: what changed and why it matters, not how it was
  implemented. One entry per change; group related tweaks into one entry.

### Keep it human

The entries should read like a person wrote them, not a model. Two hard rules:

- No em dashes or en dashes (`—`, `–`). Use a comma, a period, a colon, or parentheses instead.
- Skip the "**Bold label.** Sentence." template. Write a normal sentence. Vary the length. Drop
  filler and marketing words (vibrant, seamless, powerful, robust, rich, boasts, and so on).

### Format (the parser depends on it)

Keep the "Keep a Changelog" shape (https://keepachangelog.com/en/1.1.0/):

```
## [Unreleased]

### Added        (or Changed / Fixed / Removed / Deprecated / Security)
- One or two plain sentences describing the change from the user's side.
  Continuation lines may wrap; they fold into the same bullet.
```

- Add the `### <Section>` heading only if it doesn't already exist under `[Unreleased]`.
- Inline `**bold**` renders as bold in the app, but reach for it rarely.
- Never renumber or rewrite already-released version sections; only touch `[Unreleased]`.

Releases move `[Unreleased]` into a dated version section; see `docs/RELEASING.md`. Do that only
when cutting a release, not during normal work.

## Before finishing a change

- `pnpm check` (svelte-check, 0 errors) and `pnpm lint` must pass.
- Rust changes: `pnpm rust:fmt`, `pnpm rust:clippy` (deny warnings), `pnpm rust:test`.
- If you added/changed a Rust DTO with `#[ts(export)]`, run `pnpm typegen` to refresh bindings.
- Confirm the `CHANGELOG.md [Unreleased]` entry exists for any user-facing change.
